# Blueprint for new LonghornSilicon block repos

> **SUPERSEDED (monorepo migration, 2026-07-22).** LonghornSilicon no longer develops each
> block as its own GitHub repo. Blocks are now **block-major folders inside the `lambda`
> monorepo** (`acu/`, `kve/`, `tiu/`), developed in-place and **auto-mirrored read-only** to
> `lambda-<block>`. Ignore the "create/clone a new per-block repo" steps below (Step 0 and the
> standalone-repo names like `LonghornSilicon/kv-cache-engine`); instead add a new top-level
> block folder in the monorepo. The per-block pipeline (RTL → sim → Yosys → LibreLane sign-off,
> closed-form FF counts, two-testbench pattern, ISA/paper) is still valid — only the *repo layout*
> and *per-block CI* changed. See the repo-root `AGENTS.md`/`CLAUDE.md` and `DECISIONS.md` (2026-07-22).

**Purpose**: If you're a Claude Code session starting work on a new
LonghornSilicon block (KV Cache Engine, Token Importance Unit, Memory
Hierarchy Controller, or any future block), read this end-to-end before
writing any code. It captures the patterns that worked for the
precision controller (`attention-compute-unit`, the ACU), the
gotchas we hit, and the specific moment to migrate to a shared CI repo.

This is meant to be read cold — no prior conversation context
required. References to files in the `attention-compute-unit`
repo are examples to copy-and-adapt, not literal sources you should
import.

## What block #1 (precision controller) demonstrated

A single chip block goes through this pipeline, all reproducible in CI:

| Stage | Open tool | Cadence equivalent | Status |
|---|---|---|---|
| RTL | SystemVerilog | (same) | hand-written |
| Functional verification | iverilog testbenches | Xcelium | 253/253 pass |
| Logic synthesis | Yosys | Genus | 30 FFs, matches closed-form |
| Place & route | OpenROAD (via LibreLane) | Innovus | Sky130 signed off |
| Multi-corner STA | OpenSTA | Tempus | SS clean (+72 ps WNS) |
| Physical verification | Magic / Netgen | PVS / Calibre | DRC + LVS + antenna + IR-drop clean |
| Power | LibreLane built-in | Joules / Voltus | 330 µW at TT |
| FPGA prototype (pending) | nextpnr (iCE40) / Vivado (UltraScale+) | n/a | wait for ZCU102/104 |

The pipeline is gated in `.github/workflows/ci.yml`: zero violations
on any check fails the CI run. This is the same gate logic used in
industry tape-out methodology, just at smaller scale.

## Step-by-step: setting up a new block repo

The example file paths below come from
`attention-compute-unit` — copy them and rename for your block.

### Step 0. Create the repo

Create `LonghornSilicon/<block-name>` on GitHub (e.g.,
`LonghornSilicon/kv-cache-engine`). Public unless there's a reason to
hide it.

Clone locally:

```sh
git clone git@github.com:LonghornSilicon/<block-name>.git
cd <block-name>
```

### Step 1. Directory structure

Mirror the precision controller's layout. The exact directory tree is:

```
<block-name>/
├── analysis/                          # Python: algorithm exploration, fixed-point sim, test-vector generation
├── kernels/                           # GPU prototype if relevant (Triton, CUDA)
├── rtl/
│   ├── <block_name>.sv                # The DUT (top-level module)
│   ├── tb/
│   │   ├── tb_<block_name>.sv         # Directed testbench (synthetic inputs, integer reference)
│   │   ├── tb_realdata.sv             # Replay TB (real traces from analysis/)
│   │   └── testvectors/               # gitignored — regenerated by analysis/
│   ├── constraints/
│   │   ├── timing.sdc                 # SDC for Cadence/16FFC
│   │   └── timing_sky130.sdc          # Looser SDC for Sky130 first-pass
│   ├── genus.tcl                      # Cadence synth (parameterized over PDK paths)
│   ├── innovus.tcl                    # Cadence PnR
│   ├── mmmc.tcl                       # Multi-mode multi-corner setup
│   └── Makefile                       # Targets: sim, sim_realdata, testvectors, synth
├── openlane/
│   └── <block_name>/
│       ├── config.json                # LibreLane Sky130 config (signed-off values)
│       ├── src/<block_name>.v         # Copy of the .sv (or generated)
│       └── results/                   # GDS + PNG + signoff metrics (curated, committed)
│           └── runs/                  # gitignored — intermediate artifacts
├── paper/
│   ├── <block_name>.tex/.pdf
│   └── *.png                          # Figures
├── docs/
│   ├── chamber_setup.md               # If targeting the same Cadence chamber
│   ├── ci_setup.md                    # If using a different runner
│   └── ci_overview.md                 # Pipeline walkthrough
├── .github/workflows/ci.yml           # CI pipeline (adapted from this repo)
├── .gitignore                         # Includes runs/, build/, .aux/.bbl/.log
└── README.md                          # From-FA-style intro + arch context + CI explainer
```

### Step 2. Write the RTL with closed-form invariants

Two non-negotiables:

1. **The top module is parameterized**, with derived constants computed
   via `localparam` and `$clog2`. This is what makes the FF count
   predictable in closed form, which lets CI assert against it.
2. **Comments at the top of the .sv file state the closed-form FF count**
   so the CI assertion is greppable and the math is documentable.

Example header from `rtl/precision_controller.sv`:

```systemverilog
// FF count derivation:
//   max_acc: SCORE_WIDTH bits  →  SCORE_WIDTH FFs
//   sum_acc: SUM_W bits        →  SCORE_WIDTH + log₂N FFs
//   d_valid + d_fp16           →  2 FFs
//   Total: 2·SCORE_WIDTH + log₂N + 2
//
// For BLOCK_M = BLOCK_N = 64, SCORE_WIDTH = 8:
//   FFs = 2·8 + 12 + 2 = 30
```

The CI assertion later in `rtl-synthesis` job checks `FF_COUNT == 30`.
For your block, do the math and pick a single specific number for the
default parameters. If you can't derive it analytically, the block is
probably too big or too irregular — split it or simplify.

### Step 3. Two testbenches: directed + replay

Block #1 has two testbenches that both target the same DUT:

- **Directed TB** (`tb/tb_<block_name>.sv`): handwritten synthetic
  inputs covering boundary cases. Reference computed in software via
  the exact integer formula. ~100 cases.
- **Replay TB** (`tb/tb_realdata.sv`): reads `$readmemh` hex files
  generated by `analysis/gen_<block>_testvectors.py` from real
  workload traces (real attention scores, real KV access patterns,
  etc.). ~150 cases. This is what catches issues that synthetic tests
  miss.

Both need to pass — `make sim && make sim_realdata` from `rtl/`. The
CI greps the output for `ALL TESTS PASSED` or `Tests: N  Pass: N`.

**Make sure both testbenches print the exact pass/fail strings the
workflow greps for** (`ALL TESTS PASSED` or `Tests: <N>  Pass: <N>`).

### Step 4. Constraints (SDC) — write two, one tight one loose

- `rtl/constraints/timing.sdc` — tight, targets 16FFC at 800 MHz SS
  (1.25 ns clock period). This is what Genus + Innovus will use on the
  chamber when the PDK is provisioned.
- `rtl/constraints/timing_sky130.sdc` — loose, targets Sky130 at
  80-100 MHz (10-12.5 ns clock period). This is what LibreLane uses
  for the open signoff. Sky130 is 130 nm; it's not going to hit 800 MHz.

Both should set:
- `set_clock_uncertainty 0.1` (or 0.25 if Sky130 default; 0.1 is more
  honest for sub-50 ps CTS skew on a small block)
- `set_input_delay` / `set_output_delay` for all I/O ports
- `set_false_path -from rst*` (or however reset is named)
- `set_driving_cell` and `set_load` to model upstream/downstream

The precision controller's SDCs are good reference templates.

### Step 5. LibreLane config for Sky130

Create `openlane/<block_name>/config.json`. The key values learned the
hard way for block #1:

```json
{
    "DESIGN_NAME": "<block_name>",
    "VERILOG_FILES": "dir::src/*.v",
    "CLOCK_PORT": "clk",
    "CLOCK_PERIOD": 12.5,
    "IO_DELAY_CONSTRAINT": 5,
    "CLOCK_UNCERTAINTY_CONSTRAINT": 0.1,
    "DESIGN_IS_CORE": false,
    "FP_SIZING": "relative",
    "FP_CORE_UTIL": 50,
    "PL_TARGET_DENSITY_PCT": 60,
    "DESIGN_REPAIR_MAX_SLEW_PCT": 0,
    "GRT_DESIGN_REPAIR_MAX_SLEW_PCT": 0
}
```

**Non-obvious values to inherit:**

- `IO_DELAY_CONSTRAINT: 5` — default is 20 (% of clock period).
  Default eats 2.5 ns of budget per side at 12.5 ns clock, which is
  chip-level conservative. 5% (= 0.625 ns) is appropriate for a small
  block embedded in a larger pipeline.
- `CLOCK_UNCERTAINTY_CONSTRAINT: 0.1` — default is 0.25 ns. 0.1 ns
  matches sub-50 ps CTS skew you actually get on a small design.
- `DESIGN_REPAIR_MAX_SLEW_PCT: 0` — default is 20%; forces the repair
  tool to fix every slew violation rather than the worst 20%.

These three together turned a `-1.58 ns` SS WNS into `+0.072 ns` on
the precision controller without changing the RTL.

**SystemVerilog gotcha**: LibreLane's Yosys handles SV adequately but
expects `.v` extensions. We just copy `rtl/<block>.sv` to
`openlane/<block>/src/<block>.v` — same content, different extension.
Yosys reads it fine via `read_verilog -sv`.

### Step 6. Cadence scripts (parameterized over PDK paths)

Copy `rtl/genus.tcl`, `rtl/innovus.tcl`, `rtl/mmmc.tcl` from the
precision controller. The three things to change for your block:

1. `set TOP "<block_name>"` at the top
2. `set RTL_FILES [list <block_name>.sv]`
3. Anything block-specific in `report_*` calls (corner names usually
   stay the same since TSMC 16FFC corners are universal)

The PDK paths at the top of `genus.tcl` and `innovus.tcl` stay as
`TSMC_LIB_DIR`, `TSMC_LEF_DIR`, `TSMC_QRC_DIR` placeholders. The
chamber walkthrough (`docs/chamber_setup.md`) tells the user how to
fill these in when the PDK arrives.

### Step 7. CI workflow — copy ci.yml and adjust three things

Copy `.github/workflows/ci.yml` from the precision controller. Three
changes:

1. **Path adjustments**: every `working-directory: rtl` is correct;
   every `working-directory: openlane/precision_controller` becomes
   `working-directory: openlane/<block_name>`.
2. **FF-count assertion**: the assertion `[ "$FF_COUNT" = "30" ]`
   becomes your block's closed-form value.
3. **Sign-off metrics path**: the parser reads
   `runs/<latest>/final/metrics.json` — that part doesn't change. The
   gate logic (all 6 violation counts must be zero) stays the same.

If your block requires additional gates (e.g., the KV Cache Engine
might want a compression-ratio assertion: "the compressed output is
at most 25% of the input for the standard trace"), add them as new
Python assertions in the existing job rather than new jobs.

**The same self-hosted runner serves multiple block repos** if it's
registered at org level (see `docs/ci_setup.md` Option B). After that
migration, every block repo's `runs-on: [self-hosted, linux, x64]`
just works.

### Step 8. README — from-FA / from-precedent style

Three obligatory sections, in this order:

1. **TL;DR table**: what, why, how, hardware cost, verified, status.
2. **How this improves on the prior art**: For the precision controller
   this was "from FlashAttention to ratio gate." For the KV Cache
   Engine it'd be "from naive HBM-based KV cache to on-chip
   compressed cache." For the Token Importance Unit it'd be "from
   uniform-precision retention to mixed-precision retention."
3. **Where this fits in LonghornSilicon**: the ASCII chip block
   diagram (copy from this repo's README, highlight the new block).

Then the standard repo-content / results / reproduce / CI sections,
copying the structure from `attention-compute-unit`'s README.

### Step 9. Paper section (when applicable)

If the block warrants its own section in the LonghornSilicon paper,
follow the structure in `paper/adaptive_precision_attention.tex`:
RTL Verification → Functional → Parameter Sweep → ASAP7 →
Sky130 OpenLane End-to-End Sign-off → TSMC 16FFC Projection.

The Sky130 sign-off table is the most important addition. Hardware
numbers (area, power, WNS at 3 corners) make the section credible.

## When to extract CI to a shared repo (Option C trigger)

**Trigger condition**: you're standing up the second block repo and
about to copy `.github/workflows/ci.yml` for the second time. Stop
right there.

That's the right moment to create `LonghornSilicon/.github` (or
`LonghornSilicon/ci-templates`) with reusable workflows. The
precision controller's `ci.yml` becomes the reference; the second
block's CI calls it via `uses:`. Concrete sketch:

```yaml
# LonghornSilicon/.github/.github/workflows/openlane-signoff.yml
on:
  workflow_call:
    inputs:
      design-dir:
        required: true
        type: string
      expected-ff-count:
        required: true
        type: number
jobs:
  signoff:
    runs-on: [self-hosted, linux, x64]
    steps:
      - uses: actions/checkout@v4
      - working-directory: ${{ inputs.design-dir }}
        run: librelane --docker-no-tty --dockerized config.json
      - run: |
          python3 -c "
          import json
          m = json.load(open(f'${{ inputs.design-dir }}/runs/<latest>/final/metrics.json'))
          assert m['timing__setup_vio__count'] == 0
          # ... etc
          "
```

And in each block repo's slim `ci.yml`:

```yaml
jobs:
  sky130:
    uses: LonghornSilicon/.github/.github/workflows/openlane-signoff.yml@v1
    with:
      design-dir: openlane/kv_cache_engine
      expected-ff-count: 1248
```

Block-specific things (functional verification, the block's TB
commands) stay in each repo's `ci.yml`. Only the genuinely-shared
parts (OpenLane signoff gate, paper build pattern, maybe Yosys
synth) move to the shared repo.

**Rule of three caveat**: if block #2 turns out to be radically
different from block #1 (different testbench style, different
metrics, etc.), wait for block #3 before extracting. Two data
points isn't always enough to see the right abstraction.

## Block-specific guidance: KV Cache Engine (likely block #2)

Per the LonghornSilicon architecture notes, the KV Cache Engine is:

- 0.8 MB on-die SRAM (KV store), spilling to off-chip LPDDR5X (no eDRAM tier)
- ChannelQuant compression pipeline on writes, decompression on reads
- Goal: ~3.8× more KV context in the same LPDDR5X bandwidth

Specific deviations from the precision controller template you'll
likely need:

| Aspect | Precision controller | KV Cache Engine probably |
|---|---|---|
| Module count | 1 top module | 4–6 modules (quantizer, dequantizer, SRAM controller, eviction logic) |
| FFs | 30 (small) | 1000s (large) |
| Memory | None | SRAM macros via the PDK's memory compiler (or behavioral models in Sky130 testing) |
| Verification | Bit-exact integer reference | Compression-ratio + reconstruction-MSE checks |
| Closed-form FF count | Yes, trivially | Per sub-module, summed |
| OpenLane signoff difficulty | Easy (small) | Medium-hard (memories require LEF integration) |
| Power budget | ~30 µW | Much higher; will dominate the chip's leakage if not careful |

**Likely additional CI gates** for KV Cache Engine:

- Compression ratio ≥ 3× on a canonical trace (this is the block's
  raison d'être — gate it)
- Reconstruction MSE below some threshold (so decompression doesn't
  silently corrupt KV)
- Memory utilization ≤ 32 MB (the block's spec)

The OpenLane signoff for Sky130 will be more painful because real
on-chip memories don't exist in the Sky130 std-cell library — you'll
either generate behavioral models or use OpenRAM. For 16FFC, the
foundry's memory compiler will provide actual SRAM macros.

**Read first** before starting: the KV Cache Engine's codec is decided —
**ChannelQuant** (per-channel INT4 keys, grouped G=128 / per-token INT4
values + FP16 outlier-channel lane), and the block's RTL is already complete
through Sky130 sign-off in the `kve` block. The Memory Hierarchy
Controller routes 0.8 MB on-die SRAM ↔ off-chip LPDDR5X directly — the current
design has no separate eDRAM tier.

## Gotchas and lessons learned

These bit us during block #1; future-you should sidestep them.

### LibreLane CLI

- **Flag order matters**: `--docker-no-tty` must come *before*
  `--dockerized`. Otherwise the `-t` flag gets added anyway and
  Docker refuses to start under non-TTY conditions. The CLI help text
  warns about this in fine print.
- The Docker image tag is `3.0.3`, not `latest` — `ghcr.io/librelane/librelane:latest` 404s.
- LibreLane uses `os.execlp` to replace the Python process with
  Docker. Stdout buffers are lost across the exec, so if you need to
  see the actual `docker run` command being built, run with
  `PYTHONUNBUFFERED=1`.

### OpenLane configuration

- `IO_DELAY_CONSTRAINT` is a percentage, not a fraction (so `20` means
  20%, not 2000%). Critical because it scales with the clock period
  — increasing the clock period also increases I/O delay, making the
  budget non-monotonic in unexpected ways.
- Tool heuristics are *not* always monotonic — bumping the clock
  period 13 ns → 14 ns once made SS WNS worse. Don't sweep blindly.
- `FP_SIZING: relative` with `FP_CORE_UTIL: 50` is much better than
  picking absolute die dimensions for a first-pass design. Auto-sizing
  shortens wires and improves slew.

### Yosys

- Yosys 0.33 emits cell types like `$_SDFFE_PN0P_` with digits in the
  suffix. Regex like `DFF[A-Z_]*\s+[0-9]+` won't match (the regex
  stops at the first digit). Use `awk '/\$_S?DFF/ {sum+=$NF}'`
  instead.

### Cadence chamber (UT Austin specifically)

- The TAMU chamber's `tamu_kebo_verif` project module is **not** an
  indication of TAMU affiliation — it's a project namespace from
  prior coursework. Don't infer institution from project module names.
- Courseware accounts on UT's chamber are on a FlexLM **EXCLUDE list**
  for `Genus_Synthesis` and `Virtuoso_Digital_Implem` even though
  licenses are free. This is administrative, not technical. Email the
  CAD admin to be added.
- GPDK045 (the demo PDK preinstalled) is **analog-only** — no digital
  std cells. Use gsclib045 (separate IP library at
  `/process/hosted/gpdk/gpdk045/ip_libraries/gsclib045/`) for digital
  flows.
- The chamber has Genus + Innovus modules but the Innovus binary is
  often missing from disk (`/apps/INNOVUS<ver>/bin/` empty despite the
  module file existing). Check before assuming Innovus works.

### LaTeX

- pdflatex isn't installed by default; use `apt-get install
  texlive-latex-base texlive-latex-recommended texlive-latex-extra
  texlive-fonts-recommended texlive-bibtex-extra texlive-publishers`
  (about 800 MB).
- Standard 4-pass build: `pdflatex` → `bibtex` → `pdflatex` →
  `pdflatex`. The first pass writes the `.aux`; bibtex reads it to
  generate `.bbl`; the second and third passes resolve cross-refs.
- Clean intermediates from the repo: gitignore
  `paper/{*.aux,*.bbl,*.blg,*.log,*.out,*.toc}`.

### Git

- Don't run `git config` — the repo's commit author is set via
  per-commit env vars: `GIT_AUTHOR_NAME=themoddedcube
  GIT_AUTHOR_EMAIL=themoddedcube@gmail.com` (and matching
  `GIT_COMMITTER_*`). This preserves the existing commit history's
  author continuity.
- Force-pushing master is sometimes necessary (history scrubs) but
  always warn the user first and offer the simple-delete alternative.
- After `git-filter-repo`, the `origin` remote gets wiped — re-add it
  with `git remote add origin <url>` before pushing.

## Final checklist before declaring a new block "done"

A new block is ready to integrate into the chip when:

- [ ] RTL passes ≥200 functional tests (directed + replay)
- [ ] Yosys synth FF count matches the closed-form prediction
- [ ] LibreLane Sky130 signoff: 0 violations across all corners
- [ ] Cadence scripts (`genus.tcl`, `innovus.tcl`) parameterized and
      ready for PDK paths
- [ ] CI workflow runs all gates on every push
- [ ] README explains what / why / where-in-chip / how-to-reproduce
- [ ] Paper section drafted with hardware results table
- [ ] Block added to the chip block diagram in this repo's README
- [ ] Runner has been migrated to org-level scope (one-time, benefits
      everyone)

Once 2+ blocks are at this level, do the CI-template extraction
(Option C) so block #3 onwards is free to wire up.
